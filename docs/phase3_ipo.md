# Phase 3 — 이슈 추적 고도화 IPO 명세

> **목표**(기획안 2-4 Phase 3): 워치리스트 기반 이슈 추적, 3일 이상 반복 이슈 증분 요약, 긴급 즉시 알림.
> **전제**: Phase 2 자동화 안정화 + Step 4 요약 아카이브 축적. 공통 기반은 @docs/phase1_ipo.md 참조.

## 3-A. 워치리스트 (이슈 추적)

- **Input**: 사용자 정의 워치리스트(`config/watchlist.yaml` — 관심 기업·이슈·키워드), Step 3/4 결과, 날짜별 아카이브
- **Process**:
  1. 매일 분류·요약된 기사를 워치리스트 항목과 매칭 (`company_aliases.yaml` 정규화 활용)
  2. 매칭된 기사를 이슈 단위로 누적 추적
  3. 브리핑에 "내 워치리스트" 섹션으로 관련 신규 소식 강조 배치
- **Output**: 워치리스트별 추적 상태 + 브리핑 강조 섹션

```yaml
# config/watchlist.yaml
companies: ["삼성전자", "SK하이닉스", "TSMC"]
issues:    ["HBM 증설", "미국 수출통제"]
keywords:  ["EUV", "2나노"]
```

## 3-B. 반복 이슈 증분 요약 (3일 이상)

- **Input**: 이슈별 누적 기사(3-A), 이전 요약 아카이브
- **Process**:
  1. 동일 이슈가 3일 이상 지속되면 감지
  2. Gemini API로 이전 요약 대비 "새로 추가된 사실"만 증분 요약(delta summary) 생성
  3. 매일 전체 재요약이 아닌 변경분만 제공 → 중복 피로 방지
- **Output**: 이슈별 증분 요약 카드 (브리핑 삽입)
- **비고**: 여기서는 단순 시간순 누적. 이슈 간 연결관계를 그래프로 구조화하는 것은 Phase 4-2(이슈 지식그래프)에서 확장

## 3-C. 긴급 즉시 알림 (워치리스트 트리거)

- **Input**: 고빈도 수집분, 워치리스트 + 위험 키워드
- **Process**:
  1. 워치리스트 항목 관련 신규 기사 감지 시 정규 08:30 대기 없이 알림 후보로 승격
  2. Gemini API로 중요도·사실관계 판정 후 즉시 이메일 발송
- **Output**: 긴급 알림 이메일
- **알림 억제**: 동일 이슈 24시간 내 재발송하지 않고 정규 브리핑에서 갱신 — 알림 피로 방지
- **비고**: 통계적 급증 기반의 더 정교한 속보 감지는 Phase 4-1(이상 신호 감지)에서 확장

```json
// data/issues/tracking.json (이슈 추적 상태)
[ { "issue_id": "hbm-expansion", "first_seen": "YYYY-MM-DD",
    "article_ids": ["..."], "last_alerted_at": "ISO8601", "days_active": 5 } ]
```
