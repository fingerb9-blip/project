# 반도체 뉴스 브리핑 Agent — 전체 구현 로드맵 (Claude Code 프롬프트 모음)

이 문서는 기획안(최종본) 전체 범위를 순서대로 구현하기 위해 Claude Code 터미널에
그대로 붙여넣을 프롬프트 모음입니다. 터미널에서 레포 루트로 이동해 `claude`를
실행한 뒤, 위에서부터 순서대로 진행하세요. 각 단계가 끝나면 결과를 확인하고
다음 단계로 넘어가는 걸 권장합니다(한 번에 다 밀어붙이면 에러 원인을 찾기 어려워요).

> 전제: 이미 `claude.md` → `CLAUDE.md` 리네임 완료 (1단계 완료 상태)

### 3단계. CLAUDE.md에 전체 로드맵 반영

## Phase 1 — MVP (브리핑 한 통 받기)

### 4단계. RSS 소스 채우고 검증
```
sources/feeds.yaml을 채워줘. 이미 주석으로 있는 삼성전자 뉴스룸, SK하이닉스
뉴스룸, 디일렉 피드부터 실제로 접속해서 유효한 RSS인지 확인하고(깨졌으면
최신 URL을 찾아서 교체), 정상 확인된 것만 주석 해제해서 넣어줘. 추가로
EE Times, 전자신문, ZDNet Korea 등 반도체 뉴스를 다루는 국내외 매체 RSS도
몇 개 더 찾아서 같은 방식으로 검증 후 추가해줘. name은 반드시
config/source_tiers.yaml에 있는 표기와 일치시키고, 새로 추가한 소스는
source_tiers.yaml에도 등급을 매겨서 반영해줘. naver_search_keywords에는
"반도체", "HBM", "파운드리", "EUV" 등 핵심 키워드 5~8개를 채워줘.
```

### 5단계. config 4종 보강
```
config/company_aliases.yaml에 삼성전자·SK하이닉스 외에 TSMC, 인텔, 엔비디아,
마이크론, ASML, 어플라이드머티어리얼즈를 추가해줘. 각 기업마다 aliases(한글/
영문/종목코드)와 categories.yaml 기준에 맞는 segment를 채워줘. keywords.yaml의
화이트리스트/블랙리스트도 각 카테고리별로 3~5개씩 더 추가해서 풍부하게
만들어줘. 다 채운 뒤 CLAUDE.md의 config 스키마 설명과 실제 파일 내용이
일치하는지 확인해줘.
```

### 6단계. 의존성 설치 + .env 준비
```
requirements.txt에 있는 패키지를 설치해줘(pip install -r requirements.txt
--break-system-packages 필요하면 사용). .env.example을 복사해서 .env를
만들어줘. 단, 실제 키·비밀번호 값은 내가 직접 채울 거니까 빈 값 그대로 둬.
```
`.env`는 **직접 에디터로 열어서** 채우세요 (Claude Code 채팅창에 실제 키를
붙여넣지 마세요):
- `GEMINI_API_KEY` — [Google AI Studio](https://aistudio.google.com/apikey)
- `SMTP_USER` / `SMTP_PASSWORD` — Gmail 앱 비밀번호
- `SMTP_TO` — 브리핑 받을 주소
- `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` — [네이버 개발자센터](https://developers.naver.com/apps/#/register) (없어도 동작함)

### 7단계. 엔드투엔드 실행 & 디버깅
```
python main.py를 실행해서 Step 0~6이 끝까지 도는지 확인해줘. 에러가 나면
logs/ 아래 로그와 콘솔 출력을 보고 원인을 찾아 고쳐줘. 특히 Gemini API
스키마 오류나 429는 재시도 로직이 제대로 도는지 같이 확인해줘.
```

### 8단계. 결과 검증
```
data/archive/오늘날짜.md 내용을 보여주고, ①오늘의 핵심 ②카테고리별
③확인 필요 ④수집 상태 4개 섹션이 다 채워졌는지 확인해줘. 그리고 로그를
보고 이메일이 실제로 발송됐는지 확인해줘.
```

## Phase 1 확장 — 2-5, 2-6

### 9단계. R&D/학회·특허 소스 추가 (2-5)
```
docs/기획안_최종.md의 2-5절을 참고해서 Step 1 수집 로직을 확장해줘. IEDM,
ISSCC, VLSI 같은 반도체 학회의 프로그램 공지 RSS나 공식 페이지를 소스로
추가하고(RSS가 없으면 페이지를 주기적으로 파싱하는 방식도 검토), KIPRIS
(특허정보검색서비스) 오픈API와 USPTO 공개 검색 API를 활용해 신규 공개
특허를 수집하는 fetch_kipris_patents(), fetch_uspto_patents() 함수를
src/step1_collect.py에 추가해줘. 두 API 모두 무료 공개 API 기준으로
구현하고, 인증키가 필요하면 .env.example에 항목을 추가해줘. 수집된 항목은
기존 raw_articles와 같은 스키마({title,url,source,published_at,raw_text})로
통일해서 Step2~5 파이프라인을 그대로 탈 수 있게 해줘.
```

### 10단계. 인간 피드백 기반 필터링 자동화 (2-6)
```
docs/기획안_최종.md의 2-6절을 참고해서 인간 피드백 루프를 설계·구현해줘.
별도 공인 서버 없이 로컬에서 동작해야 하니, 브리핑 이메일 하단에 "이 기사
```

## Phase 2 — 완전 자동화 (GitHub Actions)

### 11단계. 산출물 영속화 전략
```
GitHub Actions로 넘어가기 전에 main.py 실행 결과(data/archive, logs)를
어떻게 보관할지 정하자. 실행이 끝나면 data/archive/*.md와 logs/*.log
변경분을 "자동 갱신: YYYY-MM-DD 브리핑" 메시지로 자동 git commit & push하는
로직을 step6_send.py 또는 별도 스크립트로 추가해줘. GitHub Actions 러너는
매번 새 환경이라 커밋해두지 않으면 아카이브가 날아간다는 점 감안해줘.
```

### 12단계. GitHub Actions 워크플로우 작성
```
.github/workflows/daily-briefing.yml을 만들어줘. 매일 KST 08:00에 맞는
cron으로 스케줄을 걸고(UTC 기준 전날 23:00), workflow_dispatch로 수동
실행도 가능하게 해줘. Python 설치 → requirements.txt 설치 → GEMINI_API_KEY,
SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD, SMTP_TO, NAVER_CLIENT_ID,
NAVER_CLIENT_SECRET을 GitHub Secrets에서 환경변수로 주입 → python main.py
실행 → 11단계에서 만든 커밋 단계까지 순서대로 넣어줘.
```

### 13단계. Secrets 등록 + 수동 트리거 테스트
```
방금 만든 워크플로우가 참조하는 시크릿 이름 목록을 정리해서 알려줘
(실제 값은 내가 GitHub 웹 Settings > Secrets and variables > Actions에서
직접 등록할게). 등록했다고 말하면, Actions 탭에서 daily-briefing을
workflow_dispatch로 한 번 수동 실행하는 방법을 안내해줘.
```

### 14단계. 실패 알림 검증
```
notify.py의 실패 알림 조건 중 하나를 일부러 깨뜨려서(예: SMTP_PASSWORD를
Actions Secrets에서 임시로 틀리게) 워크플로우를 돌려보고, 정말 실패 알림
메일이 오는지 확인하는 절차를 안내해줘. 확인 끝나면 원래 값으로 되돌리는
것도 잊지 말라고 알려줘.
```

## Phase 3 — 이슈 추적 인프라

### 15단계. 워치리스트 + 이슈 데이터 구조
```
docs/phase3_ipo.md(3단계에서 만든 문서)에 정리된 대로, 워치리스트 기반
이슈 추적 인프라를 만들어줘. data/issues/issues.json 저장소를 새로 만들고,
각 이슈는 {issue_id, title, company, category, first_seen, last_seen,
article_ids: []} 구조로 관리해줘. company_aliases.yaml에 등록된 기업과
watchlist.yaml(새로 만들 관심 이슈 키워드 파일)을 기준으로 Step4 요약
결과와 매칭하는 src/step_issue_track.py를 만들고 main.py에 연결해줘.
```

### 16단계. 반복 이슈 경과 요약
```
이슈가 3일 이상 지속되면 Gemini API로 "경과 요약" 한 문단을 추가로
생성해서 브리핑 하단에 "진행 중인 이슈" 섹션으로 넣어주는 로직을
step5_assemble.py에 붙여줘.
```

### 17단계. 긴급 알림 발송 인프라
```
정규 08:30 브리핑과 별개로 긴급한 사안이 생기면 즉시 메일을 보낼 수 있는
send_urgent_alert() 함수를 notify.py나 step6_send.py에 추가해줘. 같은
이슈로 24시간 내 중복 발송하지 않도록 최근 발송 이력을 
data/issues/alert_log.json에 기록해줘.
```

## Phase 4 — 획기적 확장 (권장 순서: 4-1 → 4-2 → 4-3)

### 18단계. 4-1. 이상 신호 감지 & 즉시 속보 알림
```
docs/기획안_최종.md 4-1절 스펙대로 이상 신호 감지를 구현해줘. 매시 정각
최근 1시간 신규 기사를 수집해 기업명+위험 키워드("규제","화재","셧다운",
"리콜") 언급 빈도를 최근 7일 같은 시간대 평균과 비교하는 배치 스크립트
(hourly_alert.py)를 새로 만들고, 임계치(3배) 초과 시 Gemini API로 진짜
속보인지 판별한 뒤 확정되면 17단계에서 만든 send_urgent_alert()로 즉시
발송해줘. 이 스크립트가 매시 정각 실행되도록 GitHub Actions 워크플로우도
하나 더 만들어줘.
```

### 19단계. 4-2. 이슈 지식그래프
```
docs/기획안_최종.md 4-2절 스펙대로 이슈 지식그래프를 고도화해줘. 15단계
에서 만든 이슈 매칭을 키워드 매칭에서 Gemini 기반 의미 유사도 판정으로
업그레이드하고(company_aliases.yaml로 기업명 정규화까지 활용), 브리핑에
"진행 중인 이슈 타임라인" 카드(예: "HBM 증설 이슈 — 3주간 5개 기사
누적, 최초 보도 6/15")를 추가해줘.
```

### 20단계. 4-3. 경쟁 구도 레이더
```
docs/기획안_최종.md 4-3절 스펙대로 경쟁 구도 레이더를 만들어줘. 매주
월요일에 지난 7일치 분류 결과를 기업별로 집계해서 막대그래프
(matplotlib)와 긍정/부정/중립 톤 비율, 그 주 최다 언급 이슈 3건 요약을
Gemini로 생성하는 weekly_radar.py를 만들고, 월요일 브리핑에 첨부되도록
main.py나 별도 GitHub Actions 워크플로우에 연결해줘. 최소 2~3주치 데이터가
쌓인 뒤에야 "추이" 비교가 의미 있다는 점을 참고해서, 데이터가 부족할 땐
그 사실을 브리핑에 명시하도록 처리해줘.
```
