# IPO 명세서: 반도체 업계 데일리 뉴스 브리핑 Agent

**목적**: 반도체 업계 관련 국내외 정치·사회·과학·문화 뉴스를 자동 수집·선별·요약하여 매일 오전 8시 30분에 브리핑 전달
**역할 분담 원칙**: 수집·스크리닝·요약 초안은 AI / 신뢰도 판단·맥락 해석·최종 승인은 사람

---

## [ Input ]

### 트리거
- **정기 실행**: 주말·공휴일 구분 없이 **매일(365일)** 오전 8시 00분 (KST) 스케줄러(cron) 자동 실행
  - 8시 30분 전달을 보장하기 위해 30분의 처리 버퍼 확보
  - 수집 범위는 요일과 무관하게 **최근 24시간 고정** — 매일 발송하므로 누락 구간 없음
- **수동 실행(보조)**: 긴급 이슈 발생 시 사용자가 온디맨드로 실행 가능

### 입력 데이터
| 유형 | 내용 | 형태 |
|---|---|---|
| 구조화 데이터 | 뉴스 소스: RSS 피드(국내 언론, 해외 언론, 업계 전문지 — 예: 디일렉, EE Times, SemiAnalysis 등), 뉴스 API, 정부·규제기관 공지 페이지 | API / RSS(XML) / HTML |
| 설정 파일 | 소스 목록, 토픽 프로파일(관심 키워드·카테고리 정의), 수신자 목록 | YAML/JSON |
| 상태 데이터 | 추적 중 이슈 워치리스트(진행 중인 규제 협상, 소송, 대형 투자 건 등), 전일까지 수집한 기사 ID(중복 방지용) | DB 또는 JSON |

### 입력 검증
1. **소스 접근 실패** (피드 다운, API 오류, 타임아웃): 소스별 3회 재시도 → 최종 실패 시 해당 소스 제외하고 진행하되, 브리핑 하단에 "금일 수집 실패 소스" 명시 (침묵 누락 금지)
2. **수집량 이상 감지**: 총 수집 건수가 평소 대비 임계치 이하(예: 30% 미만)면 파이프라인 오류 가능성으로 판단하고 브리핑 상단에 경고 표시 + 관리자 알림
3. **파싱 오류**: 형식이 깨진 기사(본문 추출 실패 등)는 스킵하되 로그에 기록, 헤드라인+링크만이라도 확보 가능하면 "확인 필요" 목록으로 이관
4. **설정 파일 오류**: 소스 목록/수신자 목록 로드 실패 시 실행 중단 + 즉시 관리자 알림 (잘못된 설정으로 발송하는 것 방지)

## [ Process ]

### Step 1: 수집 (Collection)
- 등록된 전 소스에서 최근 24시간 내 기사 수집 (주말 포함 매일 동일)
- 메타데이터 정규화: 제목 / 출처 / 발행시각 / URL / 본문 / 언어
- 전일 수집분과 기사 ID 대조하여 기수집 항목 제외

### Step 2: 중복 제거 (Deduplication)
- 의미 유사도 기반 클러스터링으로 "같은 사건을 다룬 여러 매체 보도"를 하나의 클러스터로 묶음
- 클러스터 대표 기사 선정 기준: 원출처 우선(공식 발표문·1차 보도) > 상세도 > 매체 신뢰도

### Step 3: 관련성 분류 (Screening)
- 각 클러스터를 반도체 업계 관련성 기준으로 **2단 분류** (리스크 관리 규칙 반영):
  - **핵심**: 확실히 중요 → 요약 대상
  - **확인 필요**: 애매하거나 약신호(weak signal) 가능성 → 헤드라인+링크 목록으로 브리핑에 포함, 사람이 훑어봄
  - **제외**: 무관 → 로그만 남기고 폐기 (주 1회 표본 점검 대상)
- 카테고리 태깅: 정치·규제 / 시장·기업 / 기술·과학 / 사회·문화

### Step 4: 요약 초안 생성 (Summarization)
- "핵심" 클러스터별 3~5문장 요약 생성
- 필수 포함 요소: **원문 링크**, 발행 시각, **[확정] / [관측·추측] 구분 태그** (뉘앙스 왜곡 방지)
- 워치리스트 이슈와 매칭되는 기사는 "추적 이슈 업데이트"로 별도 표시

### Step 5: 브리핑 조립 (Assembly)
- 문서 구조: ① 오늘의 핵심 3~5건 → ② 카테고리별 요약 → ③ 추적 이슈 업데이트 → ④ 확인 필요 헤드라인 목록 → ⑤ 수집 상태 리포트
- 워치리스트 자동 갱신: 신규 대형 이슈는 워치리스트 추가 후보로 제안 (사람 승인 후 등록)

### 판단 분기
| 조건 | 분기 동작 |
|---|---|
| 긴급 이슈 감지 (예: 주요국 수출규제 발표, 핵심 기업 중대 발표) | 브리핑 상단 "🚨 긴급" 섹션 배치. (선택) 8:30을 기다리지 않고 즉시 별도 알림 발송 |
| 뉴스량 적음 (주말·공휴일 등) | 요약 분량 자동 축소, "금일 특이사항 없음" 명시 |
| 소스 일부 수집 실패 | 정상 진행 + 실패 소스 명시 (검증 규칙 1과 연동) |
| 동일 이슈가 3일 이상 연속 등장 | 반복 요약 대신 "변화된 내용"만 증분 요약 |

### 예외 처리
1. **요약 생성 실패** (개별 항목): 해당 항목은 헤드라인+링크만 제공하고 "요약 실패" 표시 — 전체 파이프라인은 계속 진행
2. **파이프라인 전체 실패**: 8시 30분까지 브리핑 미생성 시 관리자(사용자)에게 실패 알림 발송. **절대 침묵 실패하지 않음** — "브리핑이 안 온 것"과 "뉴스가 없는 것"을 구분할 수 있어야 함
3. **타임아웃**: 8시 25분까지 미완료 시 완료된 부분까지 조립하여 발송 + "일부 처리 미완료" 표시
4. 모든 예외는 실행 로그에 기록하여 주간 점검 시 원인 분석 가능하게 함

## [ Output ]

### 산출물
- **일일 브리핑 문서 1건** (마크다운 또는 HTML 이메일 형식)
  - 구성: 오늘의 핵심 / 카테고리별 요약 / 추적 이슈 업데이트 / 확인 필요 헤드라인 / 수집 상태
  - 모든 요약 항목에 원문 링크 + [확정]/[관측] 태그 필수
- **아카이브 파일**: 날짜별 브리핑 원본 저장 (검색·회고용)
- **실행 로그**: 수집 건수, 분류 결과, 제외 목록, 오류 내역

### 전달 방식
- **주 채널**: 주말·공휴일 포함 매일 오전 8시 30분 이메일 발송 (또는 Slack/Teams 채널 게시 — 팀 공유 시)
- **보조**: 긴급 이슈는 즉시 알림 (분기 규칙에 따름)
- **저장**: 아카이브 폴더에 `YYYY-MM-DD_브리핑.md` 형식으로 자동 저장

### 검증 (품질 확인)
**자동 검증 (발송 전 Agent가 수행)**
- 모든 요약 항목에 원문 링크 존재 여부 체크 (누락 시 발송 보류)
- 항목 수 정상 범위 확인 (0건이거나 비정상적으로 많으면 경고 플래그)
- 발송 성공 여부 로그 기록

**사람 검증 (사용자가 수행 — 리스크 관리 규칙)**
- **매일**: 브리핑을 훑으며 중요 이슈는 원문 대조 (요약 왜곡 방지). "확인 필요" 목록 훑기 (약신호 누락 방지)
- **주 1회**: "제외" 처리된 기사 중 랜덤 표본 10~20건 점검 → 오탈락 발견 시 분류 프로파일 수정 (필터 과조임 방지)
- **분기 1회**: 소스 목록 점검 — 새 매체 추가, 죽은 피드 제거 (소스 편향 고착 방지)

**피드백 루프**
- 잘못 분류·요약된 항목을 사용자가 표시하면 토픽 프로파일/프롬프트에 반영하여 점진적 개선
