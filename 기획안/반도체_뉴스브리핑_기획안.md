# 기획안: 반도체 뉴스 데일리 브리핑 자동화 Agent

> 매일 아침 반도체 업계 뉴스를 자동으로 수집·선별·요약하여 08:30에 브리핑으로 받아보는 개인용 AI Agent
> 별첨: IPO 명세서, 워크플로우 다이어그램(mermaid)

---

## 1. 문제 정의 — 어떤 문제점이 있나요?

### 문제 상황
반도체 업계와 관련한 국내외 정치·사회·과학·문화 뉴스와 동향을 파악하는 일을 매일 수작업으로 하고 있다.
고통 점수를 매겨보면 **시간소모 5/5, 반복도 5/5, 스트레스 3/5** — 어렵기보다는 "지루하고 오래 걸리는" 업무로, 매일 같은 행위를 반복하지만 그 시간 대부분이 가치를 만들지 못한다.

### 프로세스 단계별 정리 (현재 수작업 기준, 6단계)

| 단계 | 내용 | 성격 |
|---|---|---|
| ① 소스 순회·수집 | 국내외 언론, 업계 전문지(디일렉, EE Times 등), 규제기관 발표를 매일 돌며 새 소식 확인 | 단순 반복 |
| ② 1차 스크리닝 | 수십~수백 건 헤드라인 중 반도체 관련만 골라내고 중복 보도 제거 | 단순 반복 |
| ③ 정독·사실 확인 | 선별된 기사를 읽고 출처 신뢰도·원문 확인 | 판단 필요 |
| ④ 맥락화·해석 | "이 뉴스가 무엇을 의미하는가" 판단 | 판단 필요 (핵심 가치) |
| ⑤ 요약·정리 | 핵심을 브리핑 형태로 정리 | 반(半)반복 |
| ⑥ 공유·후속 추적 | 정리 결과 활용, 진행 중 이슈 계속 추적 | 혼합 |

### 비효율의 핵심
전체 시간의 **60~70%가 ①~②단계(수집+스크리닝)에서 소모**된다. 그런데 이 구간에서 하는 일의 실체는 "결국 버릴 기사를 열어보고 닫는 행위"와 "같은 뉴스를 매체별로 중복해서 읽는 행위"다. 즉, **가치 생산이 0인 구간에 시간이 집중되어 있는 구조**라서 자동화 시 투자 대비 효과(ROI)가 가장 크다.

## 2. 기획 — AI를 통해 어떻게 해결할 수 있나요?

### 2-1. 규칙·패턴 분석: 어디를 AI로 대체할 수 있는가

| 단계 | AI 대체 가능성 | 근거 |
|---|---|---|
| ① 수집 | ◎ 거의 100% | RSS/API 기반의 완전한 규칙 작업 |
| ② 스크리닝 | ○ 80~90% | 의미 기반 관련성 판단·중복 제거는 LLM이 잘함 |
| ③ 신뢰도 판단 | △ 보조만 | 확정 vs 관측 구분은 사람 몫, AI는 태그로 보조 |
| ④ 맥락화·해석 | ✕ 사람 | 이 업무의 진짜 가치, 자동화 대상 아님 |
| ⑤ 요약 | ○ 초안 90% | AI 초안 + 사람 검수 구조가 최적 |
| ⑥ 전달·추적 | ○ 부분 | 발송·후속 알림은 자동화, 최종 판단은 사람 |

**설계 원칙: "AI는 초안, 사람은 판단."** 무리하게 전 단계를 자동화하지 않고, 가치가 0인 ①·②·⑤단계를 AI에 넘기고 ③·④단계는 사람이 유지한다. 이렇게 하면 체감 시간을 절반 이하로 줄이면서도 판단 품질은 지키는 구조가 된다.

### 2-2. 리스크와 대응 설계 (자동화의 부작용 통제)

| 리스크 | 대응 설계 |
|---|---|
| 약신호(weak signal) 누락 — 필터가 초기 단신을 걸러버림 | 분류를 "핵심 / **확인 필요** / 제외" 2단 구조로 하여 애매한 것은 헤드라인 목록으로 브리핑에 포함 |
| 요약 왜곡 — "검토 중"이 "확정"처럼 전달 | 모든 요약에 **원문 링크 + [확정]/[관측] 태그** 필수 |
| 침묵 실패 — 브리핑이 안 왔는데 모름 | 파이프라인 실패 시 **반드시 실패 알림** 발송 ("뉴스 없는 날"과 구분) |
| 도메인 감각 퇴화 / 소스 편향 | 매일 브리핑 훑기·원문 대조, 주 1회 제외 기사 표본 점검, 분기 1회 소스 목록 갱신 |

### 2-3. 스텝별 Breakdown — 신입에게 남기는 매뉴얼

> "매일 아침 이 순서대로만 하면 됩니다. 각 스텝에 필요한 데이터와 기술도 함께 적어둡니다."

| 스텝 | 무엇을 하는가 | 필요한 데이터 | 필요한 기술 |
|---|---|---|---|
| Step 0. 시작 | 매일 08:00(주말 포함) 자동 실행, 설정 로드. 설정이 깨져 있으면 실행 중단 + 알림 | 소스 목록·토픽 프로파일(YAML), 수신자 목록 | 스케줄러(cron) |
| Step 1. 수집 | 등록된 전 소스에서 최근 24시간 기사 수집, 메타데이터 정규화. 접속 실패 소스는 3회 재시도 후 기록 | RSS 피드 URL 목록, 기수집 기사 ID(JSON) | Python + feedparser |
| Step 2. 중복 제거 | 같은 사건을 다룬 여러 보도를 하나로 묶고 원출처 우선으로 대표 기사 선정 | Step 1 결과 | LLM 의미 유사도 판정 |
| Step 3. 분류 | 각 기사를 "핵심/확인 필요/제외"로 나누고 카테고리(정치·규제/시장·기업/기술·과학/사회·문화) 태깅 | 토픽 프로파일 | LLM 분류 |
| Step 4. 요약 | "핵심" 기사만 3~5문장 요약 + 원문 링크 + [확정]/[관측] 태그. 요약 실패 시 헤드라인+링크만 | Step 3 결과 | LLM 요약 |
| Step 5. 조립 | ①오늘의 핵심 ②카테고리별 ③확인 필요 목록 ④수집 상태 순으로 브리핑 문서 생성 | Step 4 결과, 브리핑 템플릿 | Python 템플릿 |
| Step 6. 발송·저장 | 08:30 이메일 발송, 날짜별 아카이브 저장, 실행 로그 기록. 08:30까지 미완료면 실패 알림 | SMTP 계정 | Gmail SMTP, Git |

### 2-4. 오늘 서비스화할 범위 선정

전체를 한 번에 만들지 않고 3단계로 나누되, **오늘은 Phase 1(MVP)만 만든다.**

- ✅ **오늘(Phase 1 — MVP)**: Step 1~6 중 핵심 경로만 — 수집 → 중복 제거 → 분류 → 요약 → 이메일 발송. 로컬에서 수동 실행으로 완성하여 "브리핑 한 통을 실제로 받아보는 것"이 목표
- ⏭ Phase 2: GitHub Actions로 매일 08:00 자동 실행 + 상태 저장 + 실패 알림
- ⏭ Phase 3: 워치리스트(이슈 추적)·3일 이상 반복 이슈 증분 요약·긴급 즉시 알림 등 고도화

**선정 이유**: Phase 1만으로도 시간 낭비의 60~70%를 차지하던 ①·②단계가 즉시 사라지고, Claude Code를 활용하면 하루 안에 완성 가능한 규모다. "작은 시도로 가장 큰 효과"를 내는 최소 단위가 정확히 이 범위다.

## 3. 사용할 AI 툴 및 목적 — 어떤 툴을 어떤 방식으로?

| 툴 | 목적 | 사용 방식 |
|---|---|---|
| **Claude Code** (VSCode) | 개발 도구 — 파이프라인 코드 작성 | IPO 명세서를 컨텍스트로 제공하고 Step별 모듈(수집/분류/요약/발송)을 구현·테스트. 개발 단계에서만 사용 |
| **Claude API** (Haiku 4.5) | 운영 두뇌 — Step 2~4의 중복 판정·분류·요약 | 매일 파이프라인이 API 호출. 분류·요약 작업은 Haiku급으로 충분하며, 하루 100~200건 분류 + 20~30건 요약 기준 **월 $2~5 수준** 예상 |
| **GitHub Actions** (Phase 2) | 스케줄러 — 서버 없이 매일 자동 실행 | cron으로 07:40 KST경 실행(정시 지연 대비 버퍼). 무료 사용량 내 충분. 워크플로우 실패 시 GitHub이 자동 이메일 알림 → **침묵 실패 방지를 공짜로 확보** |
| **Gmail SMTP** | 전달 채널 | 앱 비밀번호로 매일 08:30 브리핑 이메일 발송 (무료) |
| Python (feedparser 등) | 기반 기술 | RSS 수집·데이터 정규화·템플릿 조립 등 LLM이 필요 없는 규칙 작업 담당 |

**도구 선정 근거**: n8n/Make 같은 노코드 툴도 검토했으나, Claude Code가 있으면 "코딩 불필요"라는 노코드의 핵심 장점이 무의미해지고, 오히려 월 구독료·실행 제한·세밀한 로직(2단 분류, 침묵 실패 방지) 구현의 한계가 단점이 된다. 스크립트 방식은 비용이 API 호출료뿐이고, Git으로 버전 관리되며, 명세서의 모든 요구사항을 제약 없이 구현할 수 있다.

**참고 — 비용 구조**: 개발(Claude Code)은 Claude 구독에 포함되지만, 자동 파이프라인의 운영(Claude API)은 별도 API 키(종량제 크레딧)가 필요하다.

## 기대 효과

- 매일 뉴스 순회·스크리닝에 쓰던 시간의 60~70%가 즉시 제거되고, 남는 시간은 브리핑 검토와 해석(④단계)에만 사용
- 주말 포함 매일 08:30 일정한 품질의 브리핑 수신 → 동향 파악의 공백 없음
- 아카이브가 날짜별로 축적되어 추후 "특정 이슈의 흐름 정리" 같은 회고 활용 가능
