**기획안**

**반도체 뉴스 데일리 브리핑 자동화 Agent**

*매일 아침 반도체 업계 뉴스를 자동으로 수집·선별·요약하여 08:30에
브리핑으로 받아보는 개인용 AI Agent*

별첨: IPO 명세서, 워크플로우 다이어그램(mermaid)

# **1. 문제 정의 --- 어떤 문제점이 있나요?**

## **문제 상황**

반도체 업계와 관련한 국내외 정치·사회·과학·문화 뉴스와 동향을 파악하는
일을 매일 수작업으로 하고 있다. 고통 점수를 매겨보면 **시간소모 5/5,
반복도 5/5, 스트레스 3/5** --- 어렵기보다는 "지루하고 오래 걸리는"
업무로, 매일 같은 행위를 반복하지만 그 시간 대부분이 가치를 만들지
못한다.

## **비효율의 핵심**

전체 시간의 **60~70%가 ①~②단계(수집+스크리닝)에서 소모**된다. 그런데
이 구간에서 하는 일의 실체는 "결국 버릴 기사를 열어보고 닫는 행위"와
"같은 뉴스를 매체별로 중복해서 읽는 행위"다. 즉, 가치 생산이 0인
구간에 시간이 집중되어 있는 구조라서 자동화 시 투자 대비 효과(ROI)가
가장 크다.

## **2-4. 오늘 서비스화할 범위 선정**

전체를 한 번에 만들지 않고 단계로 나누되, 오늘은 Phase 1(MVP)만 만든다.

-   Phase 1 (MVP, 오늘): Step 1~6 중 핵심 경로만 --- 수집 → 중복 제거 →
    분류 → 요약 → 이메일 발송. 로컬에서 수동 실행으로 완성하여 "브리핑
    한 통을 실제로 받아보는 것"이 목표

-   Phase 2: GitHub Actions로 매일 08:00 자동 실행 + 상태 저장 + 실패
    알림

-   Phase 3: 워치리스트(이슈 추적)·3일 이상 반복 이슈 증분 요약·긴급
    즉시 알림 등 고도화

-   Phase 4 (신규 제안 --- 아래 4장 참고): 이상 신호 감지 + 이슈
    지식그래프 + 경쟁 구도 레이더

**선정 이유:** Phase 1만으로도 시간 낭비의 60~70%를 차지하던 ①·②단계가
즉시 사라지고, Claude Code를 활용하면 하루 안에 완성 가능한 규모다.
"작은 시도로 가장 큰 효과"를 내는 최소 단위가 정확히 이 범위다.

## **2-5. R&D/학술 및 특허 동향 소스 추가 (Deep-Tech 커버리지)**

**내용:** 일반 언론사나 전문지 뉴스 외에도 반도체 주요 학회(IEDM, ISSCC,
VLSI 등)의 발표 소식이나 신규 공개 특허 정보를 수집 소스(Step 1)에 일부
포함시킨다. 학회 발표 자료는 프로그램 RSS·공식 페이지를, 특허 정보는
KIPRIS·USPTO 등 공개 검색 API를 활용해 Step 1의 기존 RSS 수집 로직에
추가 소스로 연동한다.

**효과:** 단순히 기업의 비즈니스 소식에 그치지 않고 차세대 소자 물성,
패키징 기술, 설계 IP 관련 핵심 연구 트렌드를 조기에 포착할 수 있어
기술적 맥락을 파악하는 데 깊이를 더해준다. Step 3 카테고리(장비·소재,
팹리스·설계 등) 태깅에도 그대로 연결되어 별도 파이프라인 신설 없이 기존
구조에 통합된다.

## **2-6. 인간 피드백 기반 필터링 자동화 (Human-in-the-loop)**

**내용:** 매일 받아보는 브리핑 이메일 하단에 간단한 피드백 루프(예:
노이즈 신고 버튼 또는 특정 키워드 제외 링크)를 포함시킨다. 사용자가
이메일에서 특정 기사를 "노이즈"로 체크하면, 파이프라인이 이를 수신하여
keywords.yaml의 블랙리스트를 자동으로 업데이트하도록 연동한다.

**효과:** 시스템이 운영될수록 사용자 취향에 맞게 필터링 룰이 스스로
정교해진다. 2-2절의 "도메인 감각 퇴화" 대응 설계(주 1회 표본 검토)와
결합하면, 사람이 직접 검토하는 표본과 자동 반영되는 피드백이 상호
보완되어 필터 품질이 지속적으로 개선되는 구조가 된다.

# **3. 사용할 AI 툴 및 목적 --- 어떤 툴을 어떤 방식으로?**

**무료 구성 원칙:** 별도 유료 API 키 없이, 무료 API만으로 전체
파이프라인을 구성한다. 운영 두뇌 역할은 유료 Claude API 대신 *Gemini API
무료 티어(Flash/Flash-Lite)*로 대체한다. 개발 단계의 코드 작성·테스트는
계속 Claude Code(대화형)를 사용한다.

## **Gemini API 사용 시 유의점**

-   Google AI Studio에서 신용카드 등록 없이 API 키를 발급받아 사용한다.
    무료 티어는 Flash/Flash-Lite 모델만 지원하며(Pro는 유료 전용),
    프롬프트·응답이 구글 제품 개선에 활용될 수 있다는 점은 참고할 것 ---
    뉴스 요약이라 민감도는 낮은 편이다

-   무료 티어 한도(하루 약 1,500회 내외)는 이 파이프라인의 일일
    호출량(기사 수십~백여 건)보다 충분히 여유가 있어 별도 배치 최적화
    없이도 문제없다. 단, 결제(billing)를 프로젝트에 한 번 활성화하면 그
    프로젝트는 무료 티어가 사라지고 전량 과금으로 전환되므로, 테스트용
    프로젝트와 운영용 프로젝트를 분리해 두는 것을 권장

-   Phase 2(GitHub Actions로 완전 자동화)로 넘어갈 때도 Gemini API 키는
    GitHub Secrets에 등록해두면 그대로 사용 가능하다 --- claude -p
    방식과 달리 세션 토큰 만료·재인증 이슈가 없어 오히려 완전 자동화에
    더 적합하다

# **4. 획기적 확장 아이디어 --- Phase 4 (MVP 이후 차별화 제안)**

MVP(Phase 1~3)가 자리 잡은 뒤, "뉴스를 잘 요약해 주는 도구"를 넘어
"업계 흐름을 먼저 읽어주는 개인 애널리스트"로 진화시키기 위한 3가지
확장 기능이다. 기존 Step 1~6 파이프라인 위에 얹는 확장이며, 코어 로직
변경 없이 추가할 수 있다.

## **4-1. 이상 신호 감지 & 즉시 속보 알림**

**목적:** 하루 1회(08:30) 브리핑 구조를 유지하되, 특정 이슈가 평소 대비
비정상적으로 급증하면 정규 브리핑을 기다리지 않고 즉시 알림을 보낸다.
"침묵 실패"의 반대편 리스크 --- 속보를 다음 날 아침까지 놓치는 문제를
해결한다.

  -----------------------------------------------------------------------
  **항목**        **내용**
  --------------- -------------------------------------------------------
  트리거 조건     등록 소스 전체 기준, 특정 기업명 + 위험
                  키워드("규제", "화재", "셧다운", "리콜" 등)의
                  최근 1시간 언급 건수가 최근 7일 같은 시간대 평균 대비
                  임계치(예: 3배) 초과

  필요한 데이터   Step 1(수집) 결과를 시간 단위로 재사용, 소스별 키워드
                  언급 빈도의 이동평균 테이블(신규 저장 필요)

  필요한 기술     간단한 통계 로직(Python) + Gemini API로 "이 급증이
                  진짜 속보인지, 단순 재탕 보도인지" 최종 판별

  처리 흐름       ① 매시 정각 최근 1시간 신규 기사 수집 → ② 키워드·기업별
                  빈도 집계 → ③ 임계치 초과 시 Gemini API로 사실관계 확인
                  및 [확정]/[관측] 판정 → ④ 확정 속보만 즉시 이메일
                  발송, 나머지는 정규 08:30 브리핑에 포함

  알림 억제 규칙  같은 이슈로 이미 속보를 보냈으면 24시간 내 재발송하지
                  않고 정규 브리핑에서만 갱신 --- 알림 피로 방지
  -----------------------------------------------------------------------

## **4-2. 이슈 지식그래프 (Issue Knowledge Graph)**

**목적:** 단발성 기사 요약을 넘어, 동일 이슈(예: 특정 기업의 파운드리
증설)를 시간순으로 자동 연결해 "이 이슈가 언제 시작돼 어떻게 전개되고
있는가"를 보여준다. 2-1절에서 "자동화 대상 아님"으로 남겨둔 ④
맥락화·해석 단계의 사람 부담을 실제로 줄여주는 확장이다.

  -----------------------------------------------------------------------
  **항목**        **내용**
  --------------- -------------------------------------------------------
  산출물          브리핑 하단 "진행 중인 이슈 타임라인" 카드 (예: "HBM
                  증설 이슈 --- 3주간 5개 기사 누적, 최초 보도 6/15")

  필요한 데이터   Step 4(요약) 결과의 날짜별 아카이브, 이슈 ID·연결
                  관계를 저장하는 신규 테이블

  필요한 기술     Gemini API로 새 기사가 기존 이슈의 후속 보도인지 의미
                  유사도 기준으로 판정 (company_aliases.yaml로 기업명
                  정규화까지 함께 활용)

  처리 흐름       ① 매일 요약된 기사마다 기존 이슈 목록과 매칭 시도 → ②
                  매칭되면 같은 이슈 ID로 연결, 없으면 신규 이슈 생성 → ③
                  브리핑 조립 시 진행 중 이슈를 타임라인 카드로 삽입 → ④
                  이슈가 3일 이상 이어지면 "경과 요약" 한 문단을 Gemini
                  API가 추가 생성

  확장 여지       "이 이슈, 예전에도 있었나?" 질문에 아카이브를 검색해
                  답하는 간단한 RAG 챗봇으로 자연스럽게 발전 가능
  -----------------------------------------------------------------------

## **4-3. 경쟁 구도 레이더 (Competitive Radar)**

언급됐는지"를 비교하는 미니 리포트를 자동 생성한다. 매일의 개별 기사
이해를 넘어, 누적된 관점에서 업계 구도 변화를 파악하게 해준다.

  -----------------------------------------------------------------------
  **항목**        **내용**
  --------------- -------------------------------------------------------
  산출물          주 1회(예: 매주 월요일 브리핑에 첨부) 기업별 언급량
                  막대그래프 + 긍정/부정/중립 톤 비율 + 해당 주 최다 언급
                  이슈 3건 요약

  필요한 데이터   Step 3(분류) 단계에서 이미 태깅된 기사의
                  기업명·카테고리 정보를 주간 단위로 누적 집계

  필요한 기술     Python 집계·시각화(matplotlib 등 무료 라이브러리) +
                  Gemini API로 톤(긍정/부정/중립) 판정 및 이슈 요약

  처리 흐름       ① 매일 분류된 기사에 기업 태그·톤 라벨 추가 저장 → ② 주
                  1회 배치로 지난 7일치 집계 → ③ Gemini API로 "이번 주
                  이슈의 중심" 한 문단 해설 생성 → ④ 정규 브리핑
                  템플릿에 주간 섹션으로 삽입

  확장 여지       월간 누적 시 "3개월간 경쟁사 뉴스 점유율 변화" 같은
                  회고형 인사이트로 자연스럽게 발전 가능(Step 6
                  아카이브를 그대로 재활용)

# **5. 기대 효과**

-   매일 뉴스 순회·스크리닝에 쓰던 시간의 60~70%가 즉시 제거되고, 남는
    시간은 브리핑 검토와 해석(④단계)에만 사용

-   주말 포함 매일 08:30 일정한 품질의 브리핑 수신 → 동향 파악의 공백
    없음

-   아카이브가 날짜별로 축적되어 추후 "특정 이슈의 흐름 정리" 같은
    회고 활용 가능

-   (Phase 4 반영 시) 단순 요약 도구를 넘어 이슈 흐름·경쟁 구도·속보를
    함께 챙기는 개인 애널리스트로 진화
